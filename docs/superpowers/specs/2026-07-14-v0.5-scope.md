## Purpose

v0.4 made *provisioning* reliable, hands-off, and self-checking. The security review confirms
the engineering is above-average for its class — crypto is correct, no shell-injection surface,
secrets handling is clean, the system fails closed — and surfaces **one architecturally serious
finding** plus a tail of hygiene items and a product-maturity gap list.

v0.5 has one theme: **make the trust chain unimpeachable and lay the credibility floor** an
anti-censorship tool needs before it carries real at-risk users. The single load-bearing fix is
F-1 (unsigned auto-update as root, on by default) — in the one project where "trust the
distribution channel" is exactly the assumption the threat model forbids. Everything else in
this release is subordinate to closing that, then clearing the cheap hygiene tail while the
signing machinery is open.

The larger **adaptive-adversary product program** (node pooling/rotation, transport agility,
CDN diversity, censorship-resistant config distribution) is registered here for continuity but
**deferred to v0.6+**: it is a multi-release architectural push that deserves its own design
spec, not a bolt-on to a security release.

## Priority key

**P0** = blocks shipping to real at-risk users (trust-chain integrity); **P1** = security
hygiene / credibility floor; **P2** = defense-in-depth / latent-bug; **P3** = polish. (Product
roadmap items use the review's own numbering, `#1`–`#24`, and carry a Tier, not a P-level.)

## Product roadmap (review Part II) — registered, mostly deferred

The review's 24 product items separate "good provisioner" from "tool you bet your safety on."
v0.5 lands only the **Tier-1 credibility-floor** items that are cheap, high-trust-yield, and
naturally adjacent to the signing work. Tiers 2–4 are registered for v0.6+ and gated on a
dedicated adaptive-adversary design spec.

**In scope for v0.5 (Tier 1 — credibility floor):**
- **#1 Reproducible builds.** Verify `-trimpath`/`-buildvcs=false` determinism; document how a
  third party reproduces a published binary. Falls out of Workstream A (signed + reproducible is
  the pair that earns trust). *(Workstream A.)*
- **#2 `SECURITY.md`.** Explicit threat model (what RDDA does/does not defend against) + a
  disclosure policy + supported-versions. Hours of writing, large trust signal. *(Workstream D.)*
- **#20 Explicit no-logs posture, enforced in config.** Audit sing-box logging on both nodes,
  pin to the minimum, document the guarantee. Mostly config + doc; make "logs that could
  deanonymize a user" absent by construction. *(Workstream D.)*
- **#13 Off-node encrypted backup (partial).** `rdda backup` already emits an argon2id +
  XChaCha20 archive; add an optional operator-controlled **push destination** (scp/rclone-style
  target) so a lost EU node isn't a lost source of truth. Small, independent, high-value. *(Workstream D — stretch.)*

**Deferred to v0.6+ (own design spec required):**
- **Tier 2 — adaptive adversary (the spine):** #8 entry-node **pooling + rotation**, #10 real
  **client lifecycle** (revocation-now / expiry / per-client key rotation), #5 **transport
  agility** shipped-simultaneously + auto-failover, #7 **CDN diversity beyond Cloudflare**.
  Most of Tiers 3–4 hang off these primitives (the EU source-of-truth gaining a *node pool* +
  *client validity* model, and clients carrying *multiple entry endpoints*).
- **Tier 3 — responsible operation:** #21 blocking canary, #19 compromise playbook/runbook,
  #12 privacy-preserving capacity metrics, #6 active-probing-resistance self-test.
- **Tier 4 — reach & longevity:** #14 guided operator onboarding, #18 ru localization,
  #22 multi-maintainer / multi-party release signing, #23 config/state migration path,
  #15/#16 client-side health + silent failover, #17 censorship-resistant config distribution
  (research-grade — the bridges-by-email problem), #3 third-party audit, #4 binary transparency,
  #24 censorship-resistant support channel.

## Proposed v0.5 workstreams

Ordering: A is the release; B/C/D clear the tail while the signing machinery is open.

- **Workstream A — Trust chain (F-1, #1).** *The release.*
  1. **Ship first, today:** flip `rdda-update.timer` to opt-in/default-off (`install.sh` +
     `install-eu.md`), mirroring `rdda-nfqws`. One commit; removes automatic-root-RCE-by-default
     immediately, independent of the signing work.
  2. **minisign end-to-end:** release workflow generates `SHA256SUMS.minisig`; embed the
     maintainer **public key** in the binary (`//go:embed` or `const`) and in `install.sh`; add a
     shared `internal/verify` package; verify the signature over `SHA256SUMS` **before** the
     existing digest compare, in both `selfupdate.fetchRelease` and `install.sh`. A future
     `rdda verify <file>` command reuses the same package.
  3. **Reproducible builds (#1):** pin `-trimpath`, drop VCS stamping variance, document the
     reproduce recipe. Signed **and** reproducible is the pair that lets a stranger trust the
     binary without trusting the maintainer.
- **Workstream B — Transport hygiene (F-2, F-3).** Token to `Authorization` header (with the
  one-release `?token=` bridge); README verify-before-run path + published installer signature.
- **Workstream C — Latent-bug & defense-in-depth tail (F-4, F-5, F-6, F-7, F-8).** Per-IP rate
  limit on `/ru/*`; argv reload split; CI sing-box pin-diff job; RU-SSH-close confirmation;
  `InsecureSkipVerify` maintainer-warning comments.
- **Workstream D — Credibility floor (#2, #20, and #13 stretch).** `SECURITY.md`; audited
  minimal-logging posture on both nodes; optional off-node encrypted backup push.

Out of scope for v0.5 (→ v0.6+): the entire Tier-2 adaptive-adversary program and everything
gated on it (see deferred list). cosign/keyless signing (minisign chosen — single embedded key,
no Fulcio/Rekor network dependency, best fit for a solo maintainer under an RU-network threat
model; revisit cosign + multi-party signing with #22).

## minisign signing (F-1 Workstream A) — design notes

**Why minisign over cosign keyless.** cosign's transparency-log story (Fulcio/Rekor) is
excellent but adds infrastructure and *network dependencies exercised at verify time* — awkward
from inside Russia, the exact environment that must self-update reliably. minisign is a single
Ed25519 keypair: one public key embedded in the artifact, offline verification, zero external
services. Multi-party / transparency (review #22, #4) is a v0.6+ evolution, not a v0.5 blocker.

**Mechanism.**
- **Keys.** Maintainer holds the minisign secret key **off-CI** (or in an encrypted CI secret
  for the signing step only). The **public** key is committed to the repo, embedded in the
  binary at build time, and inlined in `install.sh`.
- **Release workflow.** After building the two-arch binaries and `SHA256SUMS`, sign
  `SHA256SUMS` → publish `SHA256SUMS.minisig` alongside. (Signing the sums file, which already
  binds every artifact digest, is sufficient and keeps the signature surface small.)
- **Verify path (shared `internal/verify`).** Fetch `SHA256SUMS` **and** `SHA256SUMS.minisig`;
  verify the signature against the embedded public key; **only then** run the existing
  `sha256hex(bin) == sum` compare and the atomic swap. A signature failure aborts the update
  before any file is touched — fail-closed, consistent with the rest of the system.
- **`install.sh` parity.** Same order: fetch sums + sig, `minisign -Vp <embedded-pubkey>`,
  then the existing `sha256sum -c`. Install minisign as a build dependency (or vendor a tiny
  pure-verify path) — note the bootstrap-trust caveat below.
- **Reproducible builds (#1).** `-trimpath`, `CGO_ENABLED=0`, pinned Go toolchain (already in
  CI), no VCS-dirty stamping; document `go build` flags so a third party reproduces byte-for-byte
  and checks the digest against the signed `SHA256SUMS` themselves.

**Caveats.**
- *Bootstrap trust:* `curl | bash` still fetches the installer itself over the network. The
  installer can verify **everything it downloads**, but its own integrity rests on the initial
  fetch. Mitigation: publish the installer's `.minisig` and the pubkey fingerprint out-of-band
  (README, release notes, a pinned issue) so an operator can verify the script before running —
  this is exactly review F-3.
- *Key custody = single point of coercion:* one maintainer key is one pressure point (review
  #22). Documented as an accepted v0.5 risk; multi-party signing is the v0.6+ hedge.
- *`SHA256SUMS.minisig` naming:* keep asset names stable across releases — `selfupdate`
  resolves them by fixed name (`fetchRelease`), so a rename silently breaks auto-update.

## Tests

- **Workstream A:** unit-test `internal/verify` (valid sig → pass; tampered sums → fail;
  wrong key → fail; missing `.minisig` → fail-closed). Extend `selfupdate` table tests with a
  signature seam alongside the existing checksum seam — a bad signature must abort **before**
  the binary swap (assert `.prev`/live binary untouched). `bash -n` + shellcheck the new
  `install.sh` verify block (already in CI `shell` job).
- **Workstream B:** `subserver` accepts the `Authorization` header **and** the legacy
  `?token=` during the bridge release; `health`/`pull` send the header; constant-time compare
  preserved. Doctor's control-channel probe still passes over the header path.
- **Workstream C:** rate-limiter unit test (Nth request in a window → 429); reload argv parse
  test (spaced path survives); the pin-diff CI job runs green against the current pin.
- **Release-dryrun CI:** extend the existing two-arch build job to also produce + verify a
  local minisign signature, so a broken signing path fails the pre-publish gate — not the
  operator.

## Definition of done (v0.5)

Every published release carries a **minisign signature** over its `SHA256SUMS`; `rdda update`
and `install.sh` **verify that signature against an embedded maintainer public key** and refuse
to proceed on a bad or missing signature — a substituted binary with an attacker-generated
checksum is rejected, not installed. The update timer is **opt-in / default-off** until an
operator turns it on. The RU control-channel token travels in a header, not the URL. A published
binary is **reproducible** from public source by a third party. `SECURITY.md` states the threat
model and disclosure policy. The hygiene tail (F-4/F-5/F-6/F-8) is closed with unit + CI
coverage. The adaptive-adversary product program is captured in this register and carried into a
dedicated v0.6 design spec — not started here.
