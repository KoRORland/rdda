# RDDA v0.4 — Scope & Gap Register

**Status:** draft for planning (2026-07-07)
**Builds on:** v0.3.0 (WS+mux transport on sing-box; `status`/`doctor`/`backup`/`restore`/`alert`/`update`/`heal` — CI-verified) **and the first real two-node production deployment** — RU: Moscow VPS (`msk-1`); EU: AWS `eu-west-1`; Cloudflare-fronted RU→EU data hop + subscription — brought up end-to-end 2026-07-04 → 2026-07-07.

## Purpose

v0.3 shipped the features. The first **live** deployment proved the design *works*: a granny-grade Hiddify client in-country reached the open internet through
`client → RU (REALITY) → Cloudflare → cloudflared → EU (loopback) → internet`,
with pull-sync and the health beat both flowing over Cloudflare.

But standing it up took **days of manual debugging**. The bring-up is fragile, heavily manual, full of order-dependencies and silent-failure traps, and — worst — **not self-verifying**: several failures reported success (`200`, `exit 0`, "active") while the thing was actually broken. The operator became the integration test.

v0.4 has one theme: **make provisioning a real RU + EU + Cloudflare deployment reliable, hands-off, and self-checking**, honouring the design goal that the operator is "human-in-the-loop only for adding clients or fixing issues" — not for reverse-engineering why a `200` is lying. Every gap below is drawn line-by-line from the v0.3.0 field deployment log.

## Priority key

**P0** = blocks or silently breaks a real bring-up; **P1** = hands-off / safety; **P2** = polish / docs.

## Gap register (field evidence → v0.4 disposition)

| # | Gap (observed in the field) | Pri | Disposition |
|---|---|---|---|
| G1 | **Installer aborts on flaky GitHub-from-RU.** `nfqws2` (and every release download) is re-fetched unconditionally — no skip-if-present, no timeout/retry. From Moscow the `curl` *hangs*, and `set -e` kills the install **before** the `rdda` user, `geoip-ru.srs`, and systemd units are created; re-running just re-hangs at the same step. | P0 | **Workstream A.** Idempotent downloads (skip-if-present), `curl --retry`/`--connect-timeout`/`--max-time`, resumable + mirror fallback. |
| G2 | **`rdda-nfqws` is net-negative as shipped.** The nft rule blanket-queues **all** outbound `:443` through `nfqws2 --dpi-desync=fake split2` with no fooling method → it breaks *every* TLS handshake from the RU node, including the RU→EU tunnel and `check-dest`. It had to be disabled to get anything working. | P0 | **Workstream D.** Scope the nft rule off the node's own tunnel/probe/control traffic; gate desync to actually-blocked destinations (hostlist/ipset); add `--dpi-desync-fooling=badsum`/ttl; ship **disabled by default** until a profile is validated. |
| G3 | **CF tunnel bring-up is ~12 manual, order-sensitive steps** with hand-copied tunnel IDs/creds and silent-failure traps: `install-eu.md` writes `/etc/cloudflared/config.yml` two steps before the dir is created; `cloudflared tunnel route dns` **silently no-ops** on a pre-existing DNS record, so the sub hostname kept serving a stale origin (`200 "Hello World!"`) and pull+beat were dead for hours. | P0 | **Workstream B.** `rdda cf setup`: login → create → capture ID+creds → write `config.yaml` block → render → **route DNS and verify it resolves to _this_ tunnel** → create user → install+enable service → lock firewall. Idempotent, re-runnable. |
| G4 | **RU control channel is entirely manual.** The installer never writes `/etc/rdda/pull.env`, never carries the pull/health token EU→RU, does **not** deploy `rdda-pull.{service,timer}`, and does **not** create the `/etc/sudoers.d/rdda-reload` rule the pull reload needs. `rdda-health.service` even fails outright on the missing `EnvironmentFile`, so pull-sync (G7) + health beat (G6) are silently off. | P1 | **Workstream B.** Auto-provision the RU control channel from a supplied EU sub-host + token; deploy the pull units + sudoers rule; enable the timers. |
| G5 | **doctor gives false-green.** The `control channel` / `sub endpoint` checks assert **HTTP 200 only**, so a mis-routed hostname returning `200 "Hello World!"` passed as *healthy* while the control channel was dead. There's no check that a hostname actually routes to the tunnel, and no distinction between "listening but unreachable from RU (cloud security group)" and "service down". | P1 | **Workstream C.** doctor validates the response **body** (JSON / known marker); add a "sub host routes to _this_ tunnel" probe and a "reachable from RU" probe. |
| G6 | **Permission / ownership foot-guns recur.** `chown -R rdda:rdda /etc/rdda` (per the docs) broke the sing-box service user's config read → a 2500× crash-loop; a root-run pull would re-own `singbox.json` and re-break it; a long-running `rdda-sub` returned `200` while a mis-route made its "writes" invisible. The state-dir ownership + service-user model is brittle and undocumented. | P1 | **Workstream C.** Pin & document the state-dir ownership/permission model; ensure writers own files as the service user; add a doctor self-check ("service user can read its config / write its state"); restart affected services after any render/config change during provisioning. |
| G7 | **Doc-ordering + cloud-infra gaps.** `install-eu.md` §6.4 writes `/etc/cloudflared/config.yml` before §6.6 `mkdir`s it; the EU-on-AWS **security group** must allow inbound `443` for the direct fallback (a layer *outside* `ufw`, so the node is "listening but unreachable" — a silent `i/o timeout`), and this is undocumented. | P2 | **Workstream A.** Fix the ordering; installer creates `/etc/cloudflared` for the EU role; add a "cloud firewall / security group" note (and that CF fronting removes the requirement). |

## Proposed v0.4 workstreams

Ordering: A + B remove the manual foot-guns; C makes the remaining failures self-evident; D re-enables the one feature we had to switch off.

- **Workstream A — Installer & doc robustness (G1, G7).** Idempotent, timeout/retry-guarded, resumable downloads that survive flaky GitHub-from-RU; create dirs before writing them; fix `install-eu.md` ordering; document cloud security-group requirements. The cheapest, highest-leverage fixes — G1 alone cost the first hour of every attempt.
- **Workstream B — Zero-touch Cloudflare + control channel (G3, G4).** `rdda cf setup` (EU) and a matching RU control-channel bootstrap collapse the ~15 most error-prone manual steps into one idempotent command per node, with routing verification built in so a silent `route dns` no-op can't ship a dead endpoint.
- **Workstream C — Self-verifying provisioning (G5, G6).** doctor validates **content + routing + permissions**, not just process/status; provisioning restarts affected services and ends with a `rdda doctor` gate. This is what turns "days of manual debugging" into "the tool tells you what's wrong."
- **Workstream D — nfqws2 correctness (G2).** Make the obfuscation layer safe to switch on: scoped nft rule, a fooling method, hostlist-gated, default-off, plus an integration assertion that **enabling nfqws does not break the tunnel**.

## Definition of done (v0.4)

A **clean-room bring-up** of a fresh RU + EU + Cloudflare deployment — `install.sh` on each node, then `rdda cf setup` on EU and one control-channel command on RU — reaches `rdda doctor` **0/0 on both nodes with zero manual edits** (no hand-written `pull.env`, no DNS / permission / dir-ordering fixes). Provisioning survives flaky GitHub-from-RU. A deliberately mis-routed sub host is **caught** by doctor (body/routing check), not reported green. `rdda-nfqws` can be enabled **without** breaking the tunnel. The multi-host integration harness asserts the clean-room path and the mis-route detection.

## Carried over (not v0.4 unless explicitly pulled in)

- **RU failover automation** (v0.2 G11) and **key/cert rotation** (v0.2 G12) — still "planned" per `ARCHITECTURE.md` §7; revisit in v0.5 unless a second RU node exists.
- Telegram alerting; ASN-level blocklists (CIDR-only remains the v0.2/v0.3 scope).
