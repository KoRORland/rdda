# RDDA 0.3 — W4: alerting (EU email on transitions) (design)

## 0.3 roadmap context

0.3 is the operator quality-of-life release. **W1 (health beat + `rdda status`)**, **W2 (encrypted
backup/restore)**, and **W3 (`rdda doctor`)** are shipped. This is **W4**: EU-side email alerting
that consumes W1's stored health beat and tells the operator when something breaks — and when it
recovers. It builds on W1 (the beat) and reuses the W3-style cert probe.

## Goal

The operator should be emailed when the RU node goes down, an EU service stops, or the public TLS
cert nears expiry — fired on the *transition* into a bad state (and again on recovery), never
re-spammed every tick.

## Transport (locked by the architecture)

Per `docs/ARCHITECTURE.md` §7, v1 alerting is **email via a minimal SMTP relay (`msmtp`)**, behind a
pluggable interface so Telegram can slot in later. RDDA does **not** hold SMTP credentials: it pipes
a minimal RFC-822 message to `msmtp <recipient>`; the operator configures SMTP in `/etc/msmtprc`
(or `~/.msmtprc`). The command is configurable (default `"msmtp"`).

## Conditions

Each condition has a stable **key** (for transition tracking) and a human message:

| Key | Fires when | Message |
|---|---|---|
| `ru-down` | W1 beat missing, or `now − received_at > 20 min` | "RU node down — no health beat in &lt;age&gt;" / "no health beat received yet" |
| `unit-rdda-singbox` | EU `rdda-singbox` not `active` | "EU unit rdda-singbox is not active" |
| `unit-rdda-sub` | EU `rdda-sub` not `active` | "EU unit rdda-sub is not active" |
| `unit-cloudflared` | EU `cloudflared` not `active` | "EU unit cloudflared is not active" |
| `cert-expiry` | sub-endpoint TLS cert `NotAfter` < `CertWarnDays` away | "TLS cert expires &lt;date&gt; (&lt; N days)" |

Per-unit keys so each EU service flaps independently. The `cert-expiry` probe failing (CF down /
unreachable) yields **no** cert condition — it is inconclusive, not a new alarm (cloudflared-down is
already its own condition).

## Transition engine

`<dir>/alert-state.json` holds the **last notified firing set**: `{"firing": {"<key>":"<message>"}}`.
Each run:

1. If `alert.enabled` is false → no-op.
2. Load the last firing set; compute the current conditions.
3. Build the **new** persisted set and send emails:
   - key in both last and current ⇒ still firing, already notified → keep in new set, no email.
   - key in current but not last ⇒ **newly firing** → `Send("[RDDA ALERT] …", msg)`; **add to new
     set only if the send succeeded** (a failed send is *not* recorded, so it re-fires next run).
   - key in last but not current ⇒ **resolved** → `Send("[RDDA RESOLVED] …", msg)`; on success drop
     it; **on send failure keep it in the new set** so the resolved notice retries next run.
4. Persist the new set. No transitions ⇒ no email, file unchanged.

This makes alerting idempotent and fail-soft: a `msmtp` outage delays but never drops a notification,
and a flapping condition emits exactly one FIRING and one RESOLVED per real transition.

## Components

`internal/alert` with **injectable seams** (real by default; faked in tests):

```
type Condition struct { Key, Message string }

type Engine struct {
    dir          string
    cfg          state.Config
    beatAge      func() (age time.Duration, haveBeat bool)   // from state.LoadRUHealth
    unitActive   func(unit string) bool                      // systemctl is-active (health.UnitActive)
    certNotAfter func() (time.Time, error)                   // GET sub endpoint, read leaf cert
    now          func() time.Time
    send         func(subject, body string) error            // email via msmtp (Alerter)
}
func New(dir string, cfg state.Config) *Engine
func (e *Engine) evaluate() []Condition
func (e *Engine) Run() (fired, resolved []string, err error)
func (e *Engine) SendTest() error                            // for `rdda alert --test`
```

- `Alerter` is the `send` seam. The real email impl builds `From/To/Subject/Date` + body and pipes it
  to `exec.Command(cfg.Alert.Command, cfg.Alert.Email)` stdin (default command `msmtp`).
- `internal/cli/alert.go`: `rdda alert [--test]` — `--test` calls `SendTest` (verify msmtp setup);
  plain runs `Run`. Returns an error (non-zero exit) only on a hard failure (e.g. send error on a
  fired alert), so a flaky `msmtp` surfaces in the journal.
- `internal/state/config.go`: add `type Alert struct { Enabled bool; Email string; Command string;
  CertWarnDays int }` and `Config.Alert Alert` (`yaml:"alert"`). Engine defaults empty `Command` →
  `"msmtp"` and `CertWarnDays` ≤ 0 → `14`, so a minimal `alert: {enabled: true, email: ops@x}` works.
- `deploy/systemd/rdda-alert.service` (Type=oneshot, `User=rdda`, `ExecStart=/usr/local/bin/rdda
  alert`) + `rdda-alert.timer` (`OnBootSec=2min`, `OnUnitActiveSec=5min`, `Persistent=true`). EU only.
- `install.sh` (EU role) fetches + enables `rdda-alert.timer` (harmless until `alert.enabled`). Docs
  in `install-eu.md`: configure `/etc/msmtprc`, set `alert:` in `config.yaml`, run `rdda alert --test`.

## Role / placement

Alerting runs on the **EU** node only (it holds the beat + state and sends). The timer is enabled on
the EU role. `rdda alert` reads `<dir>/config.yaml`, `<dir>/ru-health.json`, queries EU units, and
writes `<dir>/alert-state.json` — all in `/etc/rdda`, owned by the `rdda` user.

## Error handling

- `alert.enabled` false, or `alert.email` empty ⇒ `Run` no-ops (and `--test` errors with a clear
  "set alert.email" message).
- A `Send` failure on a fired/resolved key leaves that key's notification pending (per the engine
  rules) and `Run` returns the error so the journal shows it; it never aborts evaluation of the other
  conditions or touches the data plane.
- `cert-expiry` probe error ⇒ skipped (no condition), not an alert.

## Testing

**Unit (`internal/alert`):**
- `evaluate` with faked seams: stale beat ⇒ `ru-down`; a down unit ⇒ its `unit-*` key; a near-expiry
  cert ⇒ `cert-expiry`; all-healthy ⇒ no conditions; cert-probe error ⇒ no cert condition.
- Transition diff over a temp `dir`: empty→{ru-down} sends one ALERT and persists it; {ru-down}→{}
  sends one RESOLVED and clears it; steady {ru-down}→{ru-down} sends nothing.
- **Failed-send rules:** with a `send` seam that errors, a newly-firing key is NOT persisted (re-fires
  next run); a resolved key IS kept (retries). Verified by inspecting the persisted state file +
  the recorded sends.
- State-file round-trip (`alert-state.json`).
- The email Alerter (real `send`) via an injected fake `Command` (a temp script that records argv +
  stdin): asserts the recipient arg and that the message carries the Subject + body.

**CLI smoke (`internal/cli`):** `rdda alert --test --command <fake>` writes a test message; an empty
`alert.email` errors.

No integration-harness change (no SMTP in the harness; the engine is fully unit-covered).

## Out of scope (later / other WS)

Telegram (a second `Alerter` impl behind the same seam), alert throttling/escalation beyond
fire-once-per-transition, paging integrations, and alerting on RU-local conditions (RU has no SMTP /
no source of truth; it reports via the W1 beat).
