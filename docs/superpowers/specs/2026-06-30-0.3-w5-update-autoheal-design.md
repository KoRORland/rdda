# RDDA 0.3 — W5: `rdda update` + auto-heal (design)

## 0.3 roadmap context

0.3 is the operator quality-of-life release. **W1** (health beat + `rdda status`), **W2** (encrypted
backup/restore), **W3** (`rdda doctor`), and **W4** (EU email alerting) are shipped. This is **W5**,
the last workstream: keep a node current and running without the operator SSHing in — a self-update
path for the `rdda` binary and an auto-heal path for units that systemd has given up on.

## Goal

Two independent operator features:

1. **`rdda update`** — replace the running `rdda` binary with the latest published release, verified by
   checksum, with an automatic rollback if the new binary is broken. Manual by default; an opt-in,
   disabled-by-default timer can run it hands-off.
2. **auto-heal** — a small, safe timer that recovers units stuck in systemd's `failed` state (the gap
   that plain `Restart=on-failure` does not close: once a unit exhausts its start-limit, systemd stops
   retrying and leaves it failed). Enabled by default on both nodes. It never touches an active unit.

These share no logic but share the "maintain the node unattended" theme and the systemd-restart
plumbing.

## What already exists (and why these are the real gaps)

- `rdda-singbox.service` already sets `Restart=on-failure` + `RestartSec=3`, so a crashing data plane
  is already restarted. The *uncovered* case is start-limit exhaustion: after N failures in M seconds
  systemd marks the unit `failed` and stops. Auto-heal closes exactly that gap — nothing more.
- The binary is `internal/cli.Version` (ldflags-injected at release; `rdda version` prints it),
  installed to `/usr/local/bin/rdda` by `install.sh`, which already resolves the latest GitHub release
  tag and verifies `SHA256SUMS`. `rdda update` is that download-and-verify logic, plus an atomic swap
  and rollback — minus the rest of a full provision.
- `install.sh` pins `sing-box` and `nfqws2` to specific versions. **`rdda update` does not touch
  them** — bumping those is a release-level decision that ships in a new `install.sh`.

## Component 1: `rdda update`

### CLI

```
rdda update             # download latest, verify, swap, restart rdda-sub, rollback on failure
rdda update --check     # dry-run: report installed vs latest; no mutation
rdda update --rollback  # restore the previous binary from rdda.prev (manual recovery)
```

`rdda update` and `--rollback` mutate `/usr/local/bin` and restart units, so they require **root** and
error clearly otherwise. `--check` is read-only and runs as any user.

### Flow (`rdda update`, no flags)

1. Resolve the latest release tag from the GitHub API. If it equals the current `Version` → no-op,
   print `already at <tag>`, exit 0.
2. Download `rdda-linux-<arch>` and `SHA256SUMS` from that release into a temp dir; compute the
   binary's SHA-256 and compare against its line in `SHA256SUMS`. **Mismatch ⇒ error, nothing on disk
   is touched.**
3. Back up the current binary: copy `/usr/local/bin/rdda` → `/usr/local/bin/rdda.prev`.
4. **Atomic swap:** write the new bytes to `/usr/local/bin/rdda.new`, `chmod 0755`, then
   `os.Rename` over `/usr/local/bin/rdda`. The path always points at a complete, executable binary.
5. **Self-check:** exec `/usr/local/bin/rdda version`; the output must equal the target tag. Mismatch
   (or exec failure) ⇒ **rollback** (step 7).
6. **Restart** the only long-running rdda-driven unit, `rdda-sub`, so it runs the new code. (The
   oneshot units — `rdda-health`, `rdda-pull`, `rdda-alert` — pick up the new binary on their next
   timer fire; `sing-box` is a separate binary and is untouched.) Wait briefly, then check
   `systemctl is-active rdda-sub`. Not active after ~5 s ⇒ **rollback**.
7. **Rollback:** restore `rdda.prev` over `rdda` (atomic rename), restart `rdda-sub` again, and return
   a non-zero error describing why the update was reverted.
8. **Success:** keep `rdda.prev` (one generation) for a later manual `--rollback`; print
   `updated <from> -> <to>`.

### Rollback trigger (locked)

Auto-rollback fires on **(a)** the new binary failing to exec or `rdda version` not matching the target
tag, **or (b)** `rdda-sub` not `active` ~5 s after its restart. It does **not** run `rdda doctor` or any
external-state probe — doctor checks Cloudflare / cert / REALITY-dest reachability, which can FAIL for
reasons unrelated to the update and would revert a good binary on a transient outage.

Auto-rollback only catches a *broken* release (won't run, or breaks `rdda-sub`). A release that runs
fine but is subtly wrong is recovered manually with `rdda update --rollback`.

### Opt-in auto-update timer

`rdda-update.timer` is shipped **disabled** on both roles. When the operator enables it, it runs
`rdda-update.service` (oneshot, root, `ExecStart=/usr/local/bin/rdda update`) periodically with a
**randomized delay** (`OnUnitActiveSec=24h`, `RandomizedDelaySec=12h`) so opted-in nodes do not all
update in the same minute — a bad release reaches them staggered, giving the operator a window to
notice and pin. The install docs state the residual risk plainly: a subtly-bad release auto-rolls to
every opted-in node; auto-rollback only catches broken binaries; recover with `--rollback` or
re-install a pinned `--version`.

### Component (`internal/selfupdate`)

Injectable seams (real by default; faked in tests):

```go
type Updater struct {
    arch       string                                   // amd64 / arm64
    current    string                                   // current Version
    binPath    string                                   // /usr/local/bin/rdda
    resolveTag func() (string, error)                   // GitHub latest release tag
    fetch      func(tag, arch string) (bin []byte, sum string, err error) // binary bytes + its line's sha256 hex
    restart    func(unit string) error                  // systemctl restart
    isActive   func(unit string) bool                   // systemctl is-active == "active"
    runVersion func(path string) (string, error)        // exec `<path> version`, trimmed
    now        func() time.Time
}

func New(current string) *Updater
func (u *Updater) Check() (current, latest string, newer bool, err error)
func (u *Updater) Update() (from, to string, err error)
func (u *Updater) Rollback() error
```

- `prevPath()` = `binPath + ".prev"`. Swap helpers write `binPath+".new"`, chmod 0755, rename.
- Verification (`sha256(bin) == sum`) happens before any on-disk swap.
- `New` wires `resolveTag`/`fetch` to the same GitHub release URLs `install.sh` uses
  (`api.github.com/repos/<repo>/releases/latest`, `releases/download/<tag>/rdda-linux-<arch>` +
  `/SHA256SUMS`), `restart`/`isActive` to `systemctl`, and `runVersion` to `exec.Command`.

## Component 2: auto-heal

### CLI

```
rdda heal               # reset-failed + restart any unit currently in systemd 'failed' state
```

Requires root (systemctl restart). Run by the timer; safe to run by hand.

### Behavior

For each unit in a fixed candidate list `{rdda-singbox, rdda-sub, cloudflared, rdda-nfqws}`: if
`systemctl is-failed <unit>` reports exactly `failed`, run `systemctl reset-failed <unit>` then
`systemctl restart <unit>`, and record it as healed. A unit that is active, inactive, or not installed
is **not** `failed`, so it is skipped — the same candidate list is therefore correct on both roles
(EU has singbox/sub/cloudflared; RU has singbox/nfqws; absent units are skipped). One restart failure
is surfaced in the returned error but does not stop the loop over the remaining units.

Auto-heal does not notify: it logs healed units to the journal. W4 alerting observes units
independently (every ~5 min) — if heal recovers a unit before alerting's next tick, no alert fires
(silent self-heal); if heal cannot recover it (restart keeps failing), the unit stays failed and W4
alerts. Clean separation, no coupling.

### Timer

`rdda-heal.timer` is **enabled by default** on both roles (it is safe — it only touches `failed`
units). `rdda-heal.service` is oneshot, root, `ExecStart=/usr/local/bin/rdda heal`,
`OnBootSec=2min`, `OnUnitActiveSec=2min`.

### Component (`internal/heal`)

```go
type Healer struct {
    units       []string
    isFailed    func(unit string) bool   // systemctl is-failed == "failed"
    resetFailed func(unit string) error
    restart     func(unit string) error
}

func New() *Healer                       // real systemctl seams + default unit list
func (h *Healer) Run() (healed []string, err error)
```

## Privilege model

Both timers run as **root** system units (no `User=`), so they need no sudoers entries: `rdda update`
writes `/usr/local/bin` and `rdda heal` restarts arbitrary units, both of which are root operations.
This deliberately sidesteps broadening the `rdda` user's narrow `/etc/sudoers.d/rdda-reload` grant
(which exists only for `rdda pull`'s `reload-or-restart rdda-singbox`).

## install.sh + units

- Ship four new unit files on **both** roles: `rdda-heal.service`, `rdda-heal.timer`,
  `rdda-update.service`, `rdda-update.timer`.
- `systemctl enable --now rdda-heal.timer` on both roles.
- **Do not** enable `rdda-update.timer` (operator opts in).
- No sudoers change.

## Testing

**Unit — `internal/selfupdate`** (temp-dir `binPath`, faked seams):
- `Check`: latest == current ⇒ `newer=false`; latest > current ⇒ `newer=true`, both versions reported.
- `Update` up-to-date ⇒ no-op, no file writes.
- `Update` happy path ⇒ `.new` written then renamed over `binPath`, `.prev` holds the old bytes,
  `rdda-sub` restarted, returns `from,to`.
- Checksum mismatch ⇒ error, `binPath` unchanged, no `.prev`.
- Self-check fails (`runVersion` returns wrong tag) ⇒ rollback: `binPath` bytes == original, error
  returned.
- `rdda-sub` not active after restart ⇒ rollback: `binPath` restored, error returned.
- `Rollback` ⇒ `binPath` bytes == `.prev` bytes, `rdda-sub` restarted.

**Unit — `internal/heal`** (faked seams):
- A `failed` unit ⇒ `resetFailed` + `restart` called, returned in `healed`.
- An active unit ⇒ neither called, not in `healed`.
- Mixed list (one failed, one active, one absent) ⇒ only the failed one healed.
- A `restart` error on one unit ⇒ surfaced in `err`, the loop still processes the rest.

**CLI smoke — `internal/cli`:** `rdda update --check` (faked Updater seam) prints the version line;
`rdda heal` and `rdda update`/`--rollback` are registered. Root-required paths assert a clear
non-root error.

**Integration:** no new multihost assertion. `rdda update` needs two real published releases to roll
between (the harness has none); its logic is fully seam-covered, and its real-world validation is the
first `v0.2.0 → v0.3.0` upgrade. auto-heal is seam-covered; forcing a live unit into start-limit
failure in the harness is out of scope for this workstream.

## Out of scope (later / other release)

Updating `sing-box`/`nfqws` (pinned to `install.sh`); more than one generation of binary history;
GPG/signed releases (checksum-only, matching `install.sh` today); doctor-gated or functional-probe
rollback; auto-heal functional probing (restarting an `active`-but-not-serving unit). These can follow
in 0.4 if the operator wants them.
