# RDDA 0.3 — W1: Health substrate + `rdda status` (design)

## 0.3 roadmap context

0.3 is the **operator quality-of-life** release. It decomposes into five independently
shippable workstreams (each gets its own spec → plan → implementation):

| WS | Scope | Depends on |
|---|---|---|
| **W1 (this doc)** | RU→EU health beat substrate + `rdda status` | — |
| W2 | `rdda backup` / `restore` (EU state) | — |
| W3 | `rdda doctor` (active diagnostics) | W1 |
| W4 | Alerting (email on stale beat / expiry) | W1 |
| W5 | `rdda update` (self-update) + auto-heal | W1; riskiest, last |

W1 is foundational: `status`, `doctor`, and alerting all read the heartbeat it introduces.

## Goal

Give the operator a single `rdda status` that shows EU **and** RU health, without opening
any inbound management port on the RU node. The RU node has no inbound surface by design, so
EU learns RU health only from a beat the **RU sends to EU** over the existing Cloudflare/sub
channel.

## Hard constraint — the beat must not be a DPI beacon

A fixed-interval, fixed-size beat is exactly the periodic behavioral signal RDDA exists to
defeat (Signal-3). Therefore the beat is **deliberately irregular**:

- **Random interval, 1–10 min.** systemd `OnUnitActiveSec=60` + `RandomizedDelaySec=540`
  makes each firing land uniformly in [60 s, 600 s]. `Persistent=false` so a node that was
  off does **not** emit a catch-up burst (which would itself be a beacon).
- **Random payload size.** Each POST body carries a `pad` field of random length, so no two
  beats are the same size.
- **Staleness only after 20 min.** EU flags RU stale only when `now − received_at > 20 min`
  (tolerates the ≤10 min interval plus a missed beat).

The beat is HTTPS RU→Cloudflare (Signal-1 already cleared by CF); the randomized interval +
size remove the periodicity and size tells. This is **Approach 1 (a separate, randomized
beat)** — chosen to leave the working pull-sync path untouched. The stronger "carry health
inside the data tunnel / cover traffic in many directions" idea is deferred to 0.4.

## Architecture

Small, independently testable units:

### `internal/health` (new)
- `type Report struct { TS time.Time; SingboxActive, NfqwsActive bool; SingboxVersion, RDDAVersion, Pad string }` (JSON-tagged).
- `func Gather(run Runner, rddaVersion string) Report` — builds a report from local state.
  `Runner` is an injected `func(name string, args ...string) (string, error)` (default wraps
  `exec.Command`) so tests don't need real systemd. It runs `systemctl is-active rdda-singbox`
  / `rdda-nfqws` and `sing-box version`, and attaches `randomPad()`.
- `func randomPad() string` — base64 of a random number of random bytes (length uniform in
  [0, 1024]); every call differs.
- `func Send(client *http.Client, url, token string, r Report) error` — `POST url?token=…`
  with the JSON body; error unless 2xx.

### `internal/state` (new file `internal/state/health.go`)
- `type RUHealth struct { health.Report; ReceivedAt time.Time }` (embeds the report + EU clock).
- `func (s *Store) SaveRUHealth(RUHealth) error` (atomic write to `<dir>/ru-health.json`).
- `func (s *Store) LoadRUHealth() (RUHealth, bool, error)` (`ok=false` when absent).

  > No import cycle: `health` is a leaf package (imports nothing from `state`), so `state`
  > importing `health` is one-directional and safe.

### `internal/subserver` (modified)
- `POST /ru/health`: method-gate (405 otherwise), token via `?token=` (constant-time compare
  with `cfg.PullToken`, exactly as `/ru/config`), body read through `http.MaxBytesReader`
  (~64 KB, bounds the pad), JSON-decode, `store.SaveRUHealth` with an **EU-clock**
  `received_at`. 401 bad token, 400 malformed/oversized, 200 on success. Never 5xx on bad input.

### `internal/cli` (modified)
- `rdda health` (RU): `--to` (default `$RDDA_HEALTH_TO`), `--token` (default `$RDDA_PULL_TOKEN`);
  `Gather` → `Send`; exit non-zero on send failure.
- `rdda status`: **role auto-detect** — `<dir>/config.yaml` present ⇒ EU, else RU.
  - EU: `systemctl is-active` for `rdda-singbox`/`rdda-sub`/`cloudflared`; client count
    (`<dir>/clients/*.json`); RU beat from `LoadRUHealth` → `no beat yet` | `✓ last beat <age>`
    | `⚠ STALE — no beat in <age>` (threshold 20 min), plus reported units/versions.
  - RU: `systemctl is-active` for `rdda-singbox`/`rdda-nfqws`; live REALITY-dest check (reuse
    the `check-dest` extractor + dial); last-pull freshness = mtime of `<dir>/singbox.json`.

### `deploy/systemd` (new)
- `rdda-health.service` — `Type=oneshot`, `User=rdda`, `EnvironmentFile=/etc/rdda/pull.env`,
  `ExecStart=/usr/local/bin/rdda health --to ${RDDA_HEALTH_TO} --token ${RDDA_PULL_TOKEN}`.
  No sudo/caps (it only POSTs and reads `systemctl is-active`).
- `rdda-health.timer` — `OnBootSec=2min`, `OnUnitActiveSec=60`, `RandomizedDelaySec=540`,
  `Persistent=false`, `WantedBy=timers.target`.

## Data flow

```
RU  rdda-health.timer (1–10 min, jittered) → `rdda health`
      → Gather (systemctl is-active …, sing-box version, random pad)
      → POST $RDDA_HEALTH_TO?token=… ──CF──▶ EU sub server
EU  /ru/health → store.SaveRUHealth(report, received_at=now)
EU  `rdda status` → LoadRUHealth + local systemctl + client count → render
```

## Config / wiring

- **Token:** reuse `pull_token` (same RU↔EU trust boundary; the beat carries no secret).
- **`pull.env`** gains `RDDA_HEALTH_TO=https://<cf-sub-host>/ru/health` (sibling of
  `RDDA_PULL_FROM=…/ru/config`). Documented in `install-ru.md` §4.
- **`install.sh` (RU role):** fetch `rdda-health.service` + `rdda-health.timer`,
  `systemctl enable --now rdda-health.timer`. EU needs no installer change (the `/ru/health`
  route ships in the binary; the edge already proxies `/ru/` → sub server).

## Error handling

- Beat send fails (EU down / no route): `rdda health` logs to stderr, exits non-zero; the next
  jittered tick retries. Never affects the tunnel or other RU units.
- `/ru/health`: 405 non-POST, 401 bad token, 400 malformed/oversized; valid → store + 200.
- `rdda status`: missing `ru-health.json` ⇒ "no beat yet"; `systemctl`/file errors ⇒ "unknown";
  never crashes on partial state.
- Staleness uses the **EU clock** (`received_at`), never the RU-reported `ts` (don't trust a
  remote clock for liveness).

## Security

- Token-auth identical to `/ru/config`; constant-time compare. The beat exposes only that RU is
  alive (+ unit states/versions) to whoever holds the token — same trust as pull-sync.
- Body capped (64 KB) so the random pad can't be abused to fill EU disk.
- On the wire it is TLS to Cloudflare; only size/timing leak, which the randomization blunts.

## Testing

**Unit:**
- `health.Gather` with a fake `Runner` (active/inactive/missing unit permutations) → correct flags.
- `randomPad` — successive calls differ and decode as valid base64; length within bound.
- `health.Send` against an `httptest` server — token in query, 2xx → nil, non-2xx → error.
- `subserver` `/ru/health` — bad token 401, non-POST 405, oversized 400, valid → stored.
- `state.SaveRUHealth`/`LoadRUHealth` round-trip; `LoadRUHealth` absent ⇒ `ok=false`.
- `rdda status` rendering from a fixture store (EU healthy / EU stale / no-beat / RU) — asserts
  the human-readable markers; role detection by `config.yaml` presence.

**Integration (multi-host harness):**
- After provisioning, trigger **one** beat directly (`systemctl start rdda-health.service` —
  not the timer, to avoid waiting up to 10 min in CI), then assert on EU that `rdda status`
  reports the RU node healthy with a recent beat (and `ru-health.json` exists). The beat
  traverses the real RU→edge(`/ru/`)→sub-server path, so it exercises the production channel.
