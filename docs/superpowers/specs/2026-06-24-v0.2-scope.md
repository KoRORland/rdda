# RDDA v0.2 — Scope & Gap Register

**Status:** draft for planning (2026-06-24)
**Builds on:** `docs/ARCHITECTURE.md`, `basic design layout.txt`, v0.1 MVP (CI-verified) and the v0.1 Deployment & Distribution work (both complete).

## Purpose

v0.1 shipped the **data path** — a two-hop VLESS+XHTTP+REALITY tunnel (client → RU → EU)
with RU-owned split routing, proven end-to-end in CI. What it deliberately left out is
everything that makes the system *survivable under the June-2026 DPI model* and *hands-off
for the operator*. This doc registers those gaps against the original design and proposes the
v0.2 scope.

Derived from a line-by-line comparison of `basic design layout.txt` + `docs/ARCHITECTURE.md`
against the shipped v0.1 code.

## Proposed v0.2 workstreams

Ordering reflects dependencies — A unblocks the highest-value P0 items.

- **Workstream A — Cloudflare fronting (G1, G2, G7).** The keystone. Put the RU→EU tunnel and
  the subscription endpoint behind Cloudflare; activate `rdda-sub`; build `rdda-pull.timer`.
  This is what makes the design's threat model actually hold and removes manual RU touch.
- **Workstream B — Transport & routing hardening (G3, then G4/G13/G15).** XMUX tuning first
  (cheap, defeats the frequency signal). Then evaluate zapret2/`nfqws2` and a second transport
  profile through the existing `xrayconf` seam, and add externally-managed banned-CIDR ingestion
  (see below).
- **Workstream C — Self-healing & health (G6, G8-status/heal).** `rdda-health.timer`,
  `Restart=on-failure` audit, `rdda status`/`heal`.
- **Workstream D — Operator lifecycle (G8-update/backup, G9, G10, G12, G14).** `rdda update`
  (pinned upstream versions + key rotation), `rdda backup`, email alerting.

Out of scope for v0.2 (→ v0.3+): Telegram alerting, ASN-level blocklists (CIDR lists only in
v0.2), multi-RU failover automation (G11 unless a 2nd RU exists), DPI smoke-test harness.

## Externally-managed lists (G15) — design notes

The seed (L31, L33) calls for incorporating publicly-available blocked-IP lists and treating
this as a never-ending chase. So lists must be **external, refreshable, and extensible** — an
operator (or the community) adds a source by editing config, not by patching code or
redeploying.

**Mechanism:**
- **Sources are config, not code.** `config.yaml` gains a `lists:` section — an ordered set of
  named sources, each with a URL, a format (`cidr4` / `cidr6` / `domain`), an action
  (`tunnel` | `direct` | `block`), and a refresh interval. Built-in defaults ship empty so the
  feature is opt-in.
- **Pull + cache + verify.** An `rdda-lists.timer` (RU node; over Cloudflare, like `rdda-pull`)
  fetches each source, validates/normalizes entries (drop malformed CIDRs), de-dups, and
  caches to disk. A fetch failure keeps the last-good cache — lists never fail the node open.
- **Compiled into routing.** Cached entries are compiled into xray routing rules (or an
  `ipset`/`nft` set the router consumes) and merged with the existing `geoip:ru` + intl
  allowlist. Default semantics: **banned CIDRs → tunnel** (a destination RKN blocks must never
  be attempted from the RU exit), with `direct`/`block` available per source for custom lists.
- **Precedence is explicit and ordered.** List rules are evaluated in `lists:` order, then the
  built-in `geoip` rules, then the default-to-tunnel catch-all — so an operator can override a
  geoip decision with a narrower CIDR source.
- **Extensible by design.** Adding a community or private list = appending one `lists:` entry.
  Formats are pluggable (the parser dispatches on `format`), so a new list shape is a small,
  isolated addition, not a routing rewrite.

**Candidate public sources** (operator-selectable; none hardcoded):
- `antifilter.download` — aggregated RU-blocked IP/CIDR dumps (community-maintained).
- Roskomsvoboda / `zapret-info` registry mirrors — the dumped RKN registry.
- The official `vigruzki.rkn.gov.ru` registry (requires credentials; treat as an optional
  advanced source, not a default).

**Tests:** unit-test list parse/normalize/merge and rule precedence; integration-test that a
seeded banned CIDR is forced through the tunnel and that a fetch failure falls back to cache.

## Definition of done (v0.2)

A friend's Hiddify pulls its config from the EU subscription endpoint **through Cloudflare**;
the RU node receives new clients automatically via pull-sync (no manual `render ru` copy);
an unhealthy unit self-restarts and the operator gets an email on node-down/cert-expiry; and
the integration test asserts the Cloudflare-fronted path and the pull-sync, not just the
direct two-hop tunnel.
