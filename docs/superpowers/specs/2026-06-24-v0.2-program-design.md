# RDDA v0.2 — Program Design

**Status:** approved for planning (2026-06-24)
**Builds on:** `docs/superpowers/specs/2026-06-24-v0.2-scope.md` (gap register G1–G15),
`docs/ARCHITECTURE.md`, v0.1 MVP (CI-verified) + v0.1 Deployment & Distribution (both complete).
**Supersedes:** the "v0.2 disposition" column of the scope doc where they differ (this doc is the
binding plan; the scope doc remains the gap rationale).

## Purpose

v0.1 shipped the data path. v0.2 makes it **survivable under the June-2026 DPI model** and
**hands-off for the operator**. This document is the program-level design: it fixes the
cross-cutting decisions once, decomposes the work into four dependency-ordered workstreams, and
defines what "done" means. Each workstream gets its own implementation plan (via writing-plans),
built and CI-verified in order.

## Decomposition & sequencing

Four workstreams, each independently shippable and CI-verified, in dependency order:

| WS | Delivers | Gaps | Depends on |
|----|----------|------|-----------|
| **A — Cloudflare keystone** | `cloudflared` on EU (zero inbound); RU→EU tunnel via CF Tunnel; live subscription server behind CF; `rdda-pull.timer` RU config sync | G1, G2, G7 | — |
| **B — Transport & routing hardening** | XMUX tuning; externally-managed blocklist ingestion (`rdda-lists.timer`); zapret2/`nfqws2` desync; 2nd transport profile proving the seam | G3, G15, G4, G13 | A (pull/CF) |
| **C — Self-healing & health** | `rdda-health.timer`; `Restart=on-failure` audit; `rdda status` / `rdda heal` | G6, G8 (part) | A (units exist) |
| **D — Operator lifecycle** | `rdda update` (pinned upstream versions + key/cert rotation); `rdda backup`; email alerting via msmtp | G8, G9, G10, G12, G14 | C (status/alert seam) |

Ordering rationale: A is the keystone — it removes the subnet signal and the manual-RU-touch,
and unblocks B (pull/CF) and C (units to heal). D depends on C's health events for its alert
triggers.

## Cross-cutting architecture decisions

Decided once here; the workstream plans do not re-litigate them.

### 1. Cloudflare Tunnel topology (A)

The RU→EU hop goes behind Cloudflare via **Cloudflare Tunnel (`cloudflared`)**, not orange-cloud
proxying. Consequences:

- EU runs `cloudflared` as a systemd unit; tunnel credentials live in a token/credentials file.
  EU dials **outbound** to Cloudflare and exposes **no inbound port** — the exit origin is
  un-scannable.
- EU's xray XHTTP listener and `rdda-sub` rebind to **loopback only**; `cloudflared` is the sole
  ingress, routing one public hostname → xray XHTTP and one → the subscription port.
- The RU tunnel **outbound** dials the CF public hostname over **XHTTP/TLS**. REALITY is dropped
  **on this hop only** — CF terminates TLS, so REALITY cannot survive it. The **client→RU hop
  keeps VLESS+XHTTP+REALITY** (the in-country camouflage) untouched.
- EU host firewall: inbound **SSH only**; everything else is cloudflared-out.

### 2. Config schema growth

`state.Config` gains four **optional, backward-compatible** blocks (empty = feature off, no
migration):

- `cloudflare:` — tunnel hostname (data hop), subscription hostname, credentials reference.
- `lists:` — ordered, named blocklist sources: `{name, url, format, action, interval}`.
- `alerts:` — msmtp relay host/auth, recipient, enabled triggers.
- `versions:` — pinned upstream versions (xray, cloudflared, …) for `rdda update`.

One struct, additive only.

### 3. Alert seam

A single `Alerter` interface (`Notify(event)`). v0.2 ships **one** implementation: msmtp/email.
Telegram (v0.3) is a future implementation behind the same interface — no call-site changes.
Health (C) emits events; the email **sender** lands in D. C ships emitting-to-log; alerts light
up when D lands. (Confirmed acceptable ordering.)

### 4. List-ingestion model (G15)

Sources are **config, not code**. `rdda-lists.timer` (RU) fetches each source →
validates/normalizes → de-dups → caches to disk. **A fetch failure keeps the last-good cache —
lists never fail the node open.**

- **Compile target:** cached CIDRs for the `tunnel` / `direct` actions compile into **xray
  routing rules** (one routing brain, already present). The `block` action (hostile RU *outbound*
  telemetry — VK/MAX/OK) compiles into an **`nft` set** the host firewall consumes.
- **Precedence (explicit, ordered):** `lists:` order → built-in `geoip` rules →
  default-to-tunnel catch-all. A narrower CIDR source can override a geoip decision.
- **Extensibility:** add a source = append one `lists:` entry. Parser dispatches on `format`
  (`cidr4` / `cidr6` / `domain`), so a new shape is an isolated addition.
- Ships with **empty defaults** (opt-in). Candidate sources documented, none hardcoded:
  `C24Be/AS_Network_List` (ready-made `ipset`/`nft`/`.conf`/`.routes`, auto-rebuilt; also the
  source for hostile-outbound blocking), antizapret / `zapret-info` registry mirrors. Official
  `vigruzki.rkn.gov.ru` is an optional advanced source (creds/EDS), not a default.

### 5. CLI & systemd units added

- **New commands:** `rdda status` (both roles), `rdda heal` (RU), `rdda update` (both),
  `rdda backup` (EU). Existing: `init / client / render / serve` (cobra).
- **New units:** `cloudflared.service` (EU); `rdda-pull.timer`, `rdda-lists.timer`,
  `rdda-health.timer` (RU). `Restart=on-failure` audited across all `.service` units.

## Per-workstream design

### Workstream A — Cloudflare keystone (G1, G2, G7)

- **EU.** Add `cloudflared.service`; provision a named CF Tunnel: public hostname → `localhost`
  xray XHTTP, second route → `localhost` sub port. Rebind xray XHTTP + `rdda-sub` to loopback.
  Firewall to SSH-only inbound.
- **RU.** `xrayconf` renders the tunnel outbound to the CF hostname over XHTTP/TLS (REALITY
  dropped on this hop only). `rdda-pull.timer` + `rdda pull` fetch desired RU state (config + new
  clients) from EU over CF, atomically swap, reload xray. Eliminates manual `render ru` copy.
- **Subscription live (G2).** Activate `rdda-sub` behind CF; re-chown `clients/*` to the sub
  user (v0.1 M2); Hiddify pulls `vless://` over HTTPS. Out-of-band link delivery becomes the
  fallback, not the default.
- **New config:** `cloudflare:`. **New units:** `cloudflared.service` (EU), `rdda-pull.timer`
  (RU).

### Workstream B — Transport & routing hardening (G3, G15, G4, G13)

- **XMUX (G3), first — cheap, high value.** Explicit XMUX params in `xrayconf` (concurrency,
  connection reuse, `maxConcurrency` / `maxConnections`), avoiding the `scMinPostsIntervalMs`
  trap, to collapse the connection-frequency signal.
- **Blocklist ingestion (G15).** New `internal/lists` package: source registry (dispatch on
  `format`), fetch / normalize / dedup / cache, compile to xray routing (tunnel/direct) + nft set
  (block). `rdda-lists.timer` (RU). Empty defaults.
- **zapret2 / `nfqws2` (G4) — committed deliverable, not a spike.** Integrate the `nfqws2`
  egress-handshake desync helper behind the transport-profile seam. Required precursor: manually
  extract concrete `nfqws2` strategies from the `dpi-tls-june-2026` Obsidian source (a
  client-rendered SPA — render with the lightpanda headless tool). Lands as a working, tested
  feature in the v0.2 DoD.
- **2nd transport profile (G13).** Prove the swap seam with ≥1 alternate profile (e.g.
  Vision+REALITY direct) selectable by a single config value.
- **Doc fix owed:** ARCHITECTURE §6 `nfqws` → `nfqws2`.

### Workstream C — Self-healing & health (G6, G8 part)

- **`rdda-health.timer` (RU + EU).** Periodic probe (unit liveness + tunnel reachability); on
  failure → restart the unit → record → emit an alert event (via the D alert seam; degrades to
  log if alerts unconfigured).
- **`Restart=on-failure` audit.** Every `.service` gets a restart policy + backoff.
- **`rdda status`** (both roles): unit states, tunnel up/down, cert/key expiry horizon, last
  pull / list refresh. **`rdda heal`** (RU): force restart + re-pull.

### Workstream D — Operator lifecycle (G8, G9, G10, G12, G14)

- **`rdda update` (G14, G12).** Pin upstream versions in `versions:`; `update` fetches pinned
  xray/cloudflared, verifies checksum, swaps, reloads. Same command rotates REALITY keys / CF
  creds; expiry surfaces in `status` and fires an alert.
- **`rdda backup` (G10).** Tar of EU state (`config.yaml` + `clients/` + keys) to a path.
- **Email alerting (G9).** msmtp relay behind the `Alerter` seam; triggers: node-down (from C
  health), cert/key expiry. `alerts:` config block.

## Testing strategy

Per the per-minor-release integration convention.

- **Unit.** `lists` parse/normalize/dedup/merge + rule precedence; config back-compat (empty
  blocks = off); `Alerter` seam (fake impl asserts events); `xrayconf` renders XMUX params, the
  CF-tunnel outbound, and the 2nd transport profile.
- **Integration (the real bar).** Extend the existing real-systemd harness to assert:
  1. RU→EU traffic flows **through the CF Tunnel** (cloudflared up, EU has no inbound port).
  2. A Hiddify-style client pulls config from the **CF-fronted subscription** endpoint.
  3. `rdda-pull` lands a new client on RU with **no manual touch**.
  4. A seeded **banned CIDR is forced through the tunnel**; a list **fetch failure falls back to
     cache**.
  5. **`nfqws2`** desync is active on egress and a known-DPI-tripping handshake survives (assert
     the strategy is applied, not merely configured).
  6. An unhealthy unit **self-restarts** and an alert event is emitted.
- **CI.** Keep the smart-hybrid split (heavy integration gated, docs skipped). cloudflared + a CF
  Tunnel credential is the notable new cost — use a mock/loopback CF tunnel in CI with a
  documented manual real-CF verification step.

## Definition of done (v0.2)

A friend's Hiddify pulls config from the EU subscription **through Cloudflare**; the EU exit has
**zero inbound ports** (cloudflared-only); the RU node gets new clients automatically via
pull-sync; externally-managed blocklists refresh on a timer and force banned CIDRs through the
tunnel; **`nfqws2` desync is live on the RU→EU egress**; an unhealthy unit self-restarts and the
operator gets an **email** on node-down / cert-expiry; `rdda update / backup / status / heal`
exist; and integration tests assert the CF-Tunnel path, pull-sync, list-routing, and desync — not
just the direct two-hop tunnel.

## Out of scope → v0.3+

Telegram alerting; ASN-level lists (CIDR only in v0.2); multi-RU failover automation (G11);
local RU decoy (G5 — keep external-borrow REALITY); DPI smoke-test harness.
